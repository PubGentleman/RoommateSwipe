**PROMPT: Company Agent Assignment, Booking Routing & Verified Agent Badge**

---

### PART 1 — Assign Agent to Listing

In the `listings` table, add an `assigned_agent_id` column that references `users(id)`. Run this SQL in Supabase first:

```sql
ALTER TABLE public.listings
  ADD COLUMN IF NOT EXISTS assigned_agent_id UUID REFERENCES public.users(id);
```

When a company creates or edits a listing, show an **"Assign Agent"** dropdown that lists all agents under that company (query `users` where `company_id = currentCompanyId` and `role = 'host'` and `hostType = 'agent'`). The listing cannot be published without an assigned agent.

If a listing has no `assigned_agent_id`, show a warning banner on the company dashboard: **"This listing has no assigned agent and won't receive messages until one is assigned."** The listing should still be visible in the company's dashboard but marked as inactive/unpublished.

---

### PART 2 — Route Messages & Booking Requests to Assigned Agent

When a renter sends a message or booking request about a listing:

- Look up `listings.assigned_agent_id` for that listing
- Route the conversation to that agent's inbox — **not** to the company account
- The agent sees it in their normal messages tab just like an individual host would
- The agent handles all showing scheduling and booking confirmations directly

Do not send the message to the company inbox or any other agents under the company.

---

### PART 3 — Company Dashboard Overview

The company account should have a read-only overview of all activity across their agents. In the company's dashboard/home screen, add a **"Team Activity"** section that shows:

- A list of all agents under the company
- Each agent's name, number of active listings, and number of pending booking requests
- Tapping an agent expands to show their recent conversations and bookings (read-only — company cannot reply or take action on behalf of the agent)
- A filter to view: All Agents / Active Listings / Pending Requests / Confirmed Bookings

---

### PART 4 — Reassign Listings When Agent Leaves

In the company dashboard, each listing card should have a **"Reassign Agent"** option (accessible via long press or a ⋯ menu). This opens a modal to pick a different agent from the company's team. On reassign:

- Update `listings.assigned_agent_id` to the new agent's ID
- The new agent's inbox now receives all future messages for that listing
- Existing conversation history stays intact — do not delete old messages

---

### PART 5 — Verified Agent Badge in Chat & Booking Cards

When a renter is chatting with an agent (individual or company-assigned), show a small **"Verified Agent ✓"** badge next to the agent's name in:

1. The chat header (next to the agent's name/avatar at the top of the conversation)
2. The Visit Request card — below the agent's name
3. The Booking Offer card — below the agent's name
4. The listing detail screen — in the host info section next to the agent's name

Badge style: small pill, dark background, white text, blue checkmark icon. Only show if `users.hostType = 'agent'` AND `users.license_verified = true`. If the agent is under a company, also show **"[Company Name]"** in smaller text below their name on the booking card.

---

### PART 6 — Group Bookings

When a renter who is part of a group schedules a showing or accepts a booking:

- The booking/showing card in chat should show **"Group of [X]"** where X is the number of members in the group
- The host sees "Booking for a group of 3" on their end
- The booking record stores a `group_id` field (nullable) — if the renter is in a group, populate it
- All group members can see the confirmed booking/showing card in their shared group chat
- Only the **group leader** can accept or decline a booking offer — other members see it as read-only

Add `group_id UUID REFERENCES public.groups(id)` to the `bookings` table:

```sql
ALTER TABLE public.bookings
  ADD COLUMN IF NOT EXISTS group_id UUID REFERENCES public.groups(id);
```
