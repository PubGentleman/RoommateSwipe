# Replit Prompt: Fix Birthday Timezone Bug

## BUG

When a user selects November 4 as their birthday, it shows up as November 3 after saving. This is a timezone offset bug — `new Date("2000-11-04")` is interpreted as UTC midnight, which shifts back a day in US timezones (e.g., EST is UTC-5, so midnight UTC = 7PM Nov 3 local).

## ROOT CAUSE

Two files use `new Date(birthdayString)` which interprets YYYY-MM-DD strings as UTC instead of local time. The fix is to parse the string manually and use the 3-argument `new Date(year, month, day)` constructor, which creates dates in local time.

## FIX 1: `utils/zodiacUtils.ts` — Line 154

In the `calculateZodiacFromBirthday()` function, the fallback case on line 154:

**Current (broken):**
```typescript
} else {
    date = new Date(birthday);
}
```

**Fix:**
```typescript
} else {
    const [y, m, d] = birthday.split('T')[0].split('-').map(Number);
    if (y && m && d) {
      date = new Date(y, m - 1, d);
    } else {
      return undefined;
    }
}
```

## FIX 2: `utils/matchingAlgorithm.ts` — Line 260

In the `getUserAge()` function:

**Current (broken):**
```typescript
const birth = new Date(currentUser.birthday);
```

**Fix:**
```typescript
const [y, m, d] = currentUser.birthday.split('T')[0].split('-').map(Number);
const birth = new Date(y, m - 1, d);
```

## WHY THIS WORKS

- `new Date("2000-11-04")` → parsed as UTC → shifts day in local time ❌
- `new Date(2000, 10, 4)` → parsed as local time → correct day ✅

The `.split('T')[0]` handles cases where the string might come back from the database as a full ISO timestamp (`2000-11-04T00:00:00Z`).
