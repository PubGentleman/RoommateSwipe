# Prompt: Fix Profile Tab Showing Notifications Screen Instead of Profile

## Bug

When a user taps the Profile tab in the renter navigator, the Notifications screen appears instead of the Profile screen. This happens because the bell icon in `RoommatesScreen.tsx` navigates to `Profile > Notifications`, which pushes `NotificationsScreen` onto the Profile stack. React Navigation persists that stack state, so the next time the Profile tab is tapped from any other tab, it restores to `NotificationsScreen` instead of resetting to `ProfileMain`.

## Root Cause

Two places in `screens/renter/RoommatesScreen.tsx` (lines ~940 and ~1170) contain:
```tsx
(navigation as any).navigate('Profile', { screen: 'Notifications' })
```

This is the correct way to open Notifications from inside the app. The problem is not this navigation call — it's that the Profile tab's `onPress` handler in `RenterTabNavigator` does NOT reset the stack back to `ProfileMain` when the tab is tapped. It only calls `navigation.navigate(route.name)` which restores whatever screen the stack was last on.

---

## The Fix — One change in `navigation/RenterTabNavigator.tsx`

Find the `onPress` handler inside `CustomTabBar`. It currently looks like this:

```tsx
const onPress = () => {
  const event = navigation.emit({
    type: 'tabPress',
    target: route.key,
    canPreventDefault: true,
  });
  if (!event.defaultPrevented) {
    if (route.name === 'Messages') {
      navigation.navigate('Messages' as any, { screen: 'MessagesList' } as any);
    } else if (!isFocused) {
      navigation.navigate(route.name);
    }
  }
};
```

**Replace it with:**

```tsx
const onPress = () => {
  const event = navigation.emit({
    type: 'tabPress',
    target: route.key,
    canPreventDefault: true,
  });
  if (!event.defaultPrevented) {
    if (route.name === 'Messages') {
      navigation.navigate('Messages' as any, { screen: 'MessagesList' } as any);
    } else if (route.name === 'Profile') {
      navigation.navigate('Profile' as any, { screen: 'ProfileMain' } as any);
    } else if (!isFocused) {
      navigation.navigate(route.name);
    }
  }
};
```

That's the entire fix. Adding the `Profile` case before the generic `else if (!isFocused)` block ensures the Profile tab always resets to `ProfileMain` regardless of what screen the Profile stack was last showing.

---

## Why This Approach

- The bell icon navigation calls (`navigate('Profile', { screen: 'Notifications' })`) are correct and should NOT be changed — that's how in-app notification bell buttons are supposed to work.
- The fix is at the tab bar level, not at the screen level, so it handles ALL cases where the Profile stack might be mid-navigation (Notifications, PrivacySecurity, Plans, etc.) — pressing the tab always returns to the top of the Profile screen.
- This mirrors the same pattern already used for the Messages tab (`{ screen: 'MessagesList' }`), so it's consistent.

---

## Also Apply the Same Fix to `HostTabNavigator.tsx`

The host navigator has the same potential issue — `HostDashboardScreen.tsx` has a bell icon that also navigates to `Profile > Notifications`:

```tsx
parent.navigate('Profile', { screen: 'Notifications' });
```

In `navigation/HostTabNavigator.tsx`, find the `onPress` handler inside `HostCustomTabBar` and apply the same fix:

```tsx
if (route.name === 'Messages') {
  navigation.navigate('Messages' as any, { screen: 'MessagesList' } as any);
} else if (route.name === 'Profile') {
  navigation.navigate('Profile' as any, { screen: 'ProfileMain' } as any);
} else if (!isFocused) {
  navigation.navigate(route.name);
}
```

---

## What NOT to Change

- `RoommatesScreen.tsx` — do not touch the bell icon navigation calls
- `HostDashboardScreen.tsx` — do not touch the bell icon navigation call
- `ProfileStackNavigator.tsx` — no changes needed
- `ProfileScreen.tsx` — no changes needed
- Any screen that calls `navigation.navigate('Notifications')` directly — those are fine as-is
