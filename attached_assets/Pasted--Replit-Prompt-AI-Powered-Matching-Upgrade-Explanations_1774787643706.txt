# Replit Prompt: AI-Powered Matching Upgrade — Explanations, Re-Ranking, Free-Text Preferences & Host Matchmaker

## GOAL

Transform the matching system from a rigid point calculator into an AI-augmented matchmaker. Four new features layered on top of the existing 100-point deterministic algorithm:

1. **AI Match Explanations** — Claude generates natural language compatibility narratives for every match ("You both love cooking and keep early hours. Heads up — they prefer overnight guests while you don't.")
2. **AI Deck Re-Ranking** — After the deterministic algorithm scores candidates, Claude re-ranks the top batch by analyzing full profiles holistically, catching nuances the point system misses
3. **Free-Text "Ideal Roommate" Field** — Users describe what they want in plain English; Claude parses it into structured signals AND uses it for deeper AI matching
4. **AI Host Matchmaker** — For agents/companies on paid tiers, Claude proactively recommends renters and groups for their listings with explanations

## WHAT EXISTS TODAY

### Matching Algorithm (`utils/matchingAlgorithm.ts` — 1010 lines)

100-point weighted scoring across 16 categories:

**High Priority (62 pts):**
- Location (16 pts) — neighborhood/distance matching
- Budget (12 pts) — budget variance scoring
- Sleep Schedule (12 pts) — schedule compatibility
- Cleanliness (12 pts) — cleanliness level diff
- Smoking/Substances (10 pts) — compatibility tiers
- Age (8 pts) — life stage proximity (replaced by move-in timeline below in priority)
- Move-in Timeline (4 pts) — date alignment

**Medium Priority (24 pts):**
- Work Location (6 pts) — remote/office/hybrid
- Guest Policy (6 pts) — overnight guests tolerance
- Noise Tolerance (4 pts) — noise level matching
- Pets (4 pts) — pet ownership/tolerance

**Lower Priority (14 pts):**
- Personality Compatibility (~15% weight) — 5-question quiz results
- Roommate Relationship (2 pts) — social expectations
- Shared Expenses (2 pts) — split preferences
- Lifestyle/Interest Tags (2 pts) — shared interests with conflict penalties
- Zodiac (2 pts) — optional element-based compatibility

**Dealbreaker System:** Hard blocks for `no_smokers`, `no_pets`, `no_overnight_guests`, etc. — these skip scoring entirely and return 0.

### Existing AI Edge Functions (Supabase Deno Functions)

All use Anthropic API — `claude-haiku-4-5-20251001` for fast tasks, `claude-sonnet-4-5` for complex reasoning.

1. **`ai-assistant/index.ts`** (27KB) — Streaming chat with full profile context, AI memory extraction, profile field collection. Daily message limits by plan.
2. **`agent-pair-group/index.ts`** (9.7KB) — AI group recommendations for agents. Takes 2+ renters, recommends best group for a listing. Returns JSON with confidence, reasoning, alternatives.
3. **`calculate-match-scores/index.ts`** (6.3KB) — Pre-calculates simplified scores (budget, sleep, cleanliness, smoking, move-in, pets = 100pts). Runs after swipes. Only re-scores every 24h.
4. **`ai-ask-about-profile/index.ts`** (14.7KB) — AI asks contextual questions to fill profile gaps.
5. **`ai-neighborhood-info/index.ts`** (14KB) — Neighborhood intelligence (transit, vibes, safety, cost).

### AI Memory System

`user_ai_memory` table stores extracted context from conversations:
- Budget stated, move-in timeline, dealbreakers, must-haves
- Preferred neighborhoods, transit preferences
- Work schedule, social preferences
- Free-form memory notes from past conversations

### Discover Service (`services/discoverService.ts` — 335 lines)

Builds swipe deck with filters (budget, room type, compatibility). Applies activity decay scoring and boost rotation. Does NOT use AI for ordering.

### Pre-Calculated Scores (`calculate-match-scores`)

Simpler 100-point score (budget 25, sleep 20, cleanliness 20, smoking 15, move-in 10, pets 10) stored in a scores table. Candidates not scored in 24h get re-scored. Triggers `generate-group-suggestions`.

## FEATURE 1: AI MATCH EXPLANATIONS

### What It Does

When a user views a match (swipe card, match detail screen, group member list), they see a short AI-generated narrative explaining WHY they're compatible — not just a number.

### New Edge Function: `generate-match-explanation/index.ts`

Create `supabase/functions/generate-match-explanation/index.ts`:

```typescript
// Input: two user IDs + their pre-calculated score breakdown
// Output: { summary: string, highlights: string[], warnings: string[], confidence: string }

// Example output:
{
  summary: "Strong match — you're both clean, budget-aligned, and want the same Brooklyn neighborhoods.",
  highlights: [
    "Both early risers with similar sleep schedules",
    "Budget within 5% of each other ($1,400 vs $1,350)",
    "You both picked Williamsburg and Bushwick"
  ],
  warnings: [
    "They allow overnight guests frequently — you prefer rarely"
  ],
  confidence: "high" // high, medium, low
}
```

**Claude Prompt Structure:**
```
You are a roommate compatibility analyst for a housing app. Given two user profiles and their compatibility score breakdown, generate a brief, friendly explanation of their match.

USER A PROFILE:
- Name: ${userA.first_name}
- Age: ${userA.age}
- Budget: $${userA.budget_max}/month
- Neighborhoods: ${userA.preferred_neighborhoods?.join(', ')}
- Sleep: ${userA.sleep_schedule}
- Cleanliness: ${userA.cleanliness}/10
- Smoking: ${userA.smoking}
- Pets: ${userA.pets}
- Work: ${userA.work_location}
- Guest Policy: ${userA.guest_policy}
- Move-in: ${userA.move_in_date}
- Bio/Ideal Roommate: ${userA.ideal_roommate_text || 'not set'}
- Interests: ${userA.interest_tags?.join(', ')}
- Dealbreakers: ${userA.dealbreakers?.join(', ')}

USER B PROFILE:
[same fields]

SCORE BREAKDOWN:
- Overall: ${score.total}/100
- Location: ${score.location}/16
- Budget: ${score.budget}/12
- Sleep: ${score.sleep}/12
- Cleanliness: ${score.cleanliness}/12
- Smoking: ${score.smoking}/10
- Age: ${score.age}/8
- Work: ${score.work}/6
- Guest: ${score.guest}/6
- Noise: ${score.noise}/4
- Pets: ${score.pets}/4
- Timeline: ${score.timeline}/4
- Personality: ${score.personality}/15%

RULES:
1. Summary must be 1 sentence, max 20 words. Start with the match strength ("Strong match", "Good match", "Decent match", "Mixed match").
2. List 2-4 highlights — specific things they have in common. Reference actual values, not vague statements.
3. List 0-2 warnings — potential friction points. Be honest but kind. Only include if there's a real conflict (score < 50% in a category).
4. Confidence is "high" if total > 75, "medium" if 50-75, "low" if < 50.
5. Never mention the numerical score — speak in human terms.
6. If ideal_roommate_text exists for either user, look for alignment or conflicts between their free-text descriptions.

Return valid JSON only.
```

**Use `claude-haiku-4-5-20251001`** for this — it's fast and cheap enough to run at scale.

### Caching & Storage

Create a new table to cache explanations:

```sql
CREATE TABLE IF NOT EXISTS public.match_explanations (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_a_id UUID REFERENCES public.users(id) ON DELETE CASCADE,
  user_b_id UUID REFERENCES public.users(id) ON DELETE CASCADE,
  summary TEXT NOT NULL,
  highlights JSONB NOT NULL DEFAULT '[]',
  warnings JSONB NOT NULL DEFAULT '[]',
  confidence TEXT NOT NULL DEFAULT 'medium',
  score_snapshot JSONB, -- the score breakdown at time of generation
  created_at TIMESTAMPTZ DEFAULT NOW(),
  expires_at TIMESTAMPTZ DEFAULT NOW() + INTERVAL '7 days',
  UNIQUE(user_a_id, user_b_id)
);

CREATE INDEX idx_match_explanations_users ON public.match_explanations(user_a_id, user_b_id);
CREATE INDEX idx_match_explanations_expiry ON public.match_explanations(expires_at);
```

**Expiry logic:** Regenerate if:
- Explanation is > 7 days old
- Either user updated their profile since the explanation was generated
- The score changed by more than 10 points

### When to Generate

- **Eager generation (background):** When `calculate-match-scores` runs, trigger explanation generation for any match scoring > 60. Store in cache.
- **Lazy generation (on-demand):** If a user views a match card and no cached explanation exists, call the edge function in real-time and cache the result.
- **Never generate** for matches below 40 — not worth the AI cost.

- **Plus renters:** See summary + highlights

## FEATURE 2: AI DECK RE-RANKING

### What It Does

After the deterministic algorithm scores all candidates, the top 30 are passed to Claude for holistic re-ranking. Claude reads full profiles side by side and reorders based on nuances the point system can't capture.

### New Edge Function: `ai-rerank-deck/index.ts`

Create `supabase/functions/ai-rerank-deck/index.ts`:

```typescript
// Input: user profile + array of 30 candidate profiles with scores
// Output: reordered array of candidate IDs with AI reasoning

// The function:
// 1. Takes the user's full profile (including ideal_roommate_text, ai_memory, interest_tags)
// 2. Takes top 30 candidates from deterministic scoring
// 3. Sends to Claude for holistic re-ranking
// 4. Returns new order + brief reason for top 5 promotions/demotions
```

**Claude Prompt Structure:**
```
You are an expert roommate matchmaker. A user is looking for a roommate. You've been given their full profile and 30 potential matches pre-scored by an algorithm.

Your job: re-rank these 30 candidates by TRUE compatibility, catching things the algorithm misses.

THE USER:
- Name: ${user.first_name}, Age: ${user.age}
- Bio: ${user.bio}
- Ideal Roommate Description: ${user.ideal_roommate_text || 'not provided'}
- AI Memory Notes: ${user.ai_memory_notes || 'none'}
- Interests: ${user.interest_tags?.join(', ')}
- Lifestyle: sleep ${user.sleep_schedule}, cleanliness ${user.cleanliness}/10, ${user.smoking} smoker
- Budget: $${user.budget_max}, Move-in: ${user.move_in_date}
- Neighborhoods: ${user.preferred_neighborhoods?.join(', ')}
- Dealbreakers: ${user.dealbreakers?.join(', ')}
- Work: ${user.work_location}
- Guest policy: ${user.guest_policy}
- Personality quiz results: ${JSON.stringify(user.personality_answers)}

CANDIDATES (ranked by algorithm score):
${candidates.map((c, i) => `
#${i+1} [ID: ${c.id}] Score: ${c.score}/100
  Name: ${c.first_name}, Age: ${c.age}
  Bio: ${c.bio}
  Ideal Roommate: ${c.ideal_roommate_text || 'not provided'}
  Interests: ${c.interest_tags?.join(', ')}
  Sleep: ${c.sleep_schedule}, Clean: ${c.cleanliness}/10, Smoking: ${c.smoking}
  Budget: $${c.budget_max}, Move-in: ${c.move_in_date}
  Neighborhoods: ${c.preferred_neighborhoods?.join(', ')}
  Work: ${c.work_location}, Guests: ${c.guest_policy}
`).join('\n')}

INSTRUCTIONS:
1. Re-rank all 30 candidates from best to worst fit for this specific user.
2. Consider:
   - Lifestyle alignment beyond what scores capture (e.g., two creative professionals vs. a creative + corporate worker)
   - Bio/ideal roommate text compatibility — do their descriptions align?
   - Interest overlap depth — sharing 1 niche hobby is more meaningful than sharing 3 generic ones
   - Potential personality clashes hidden in the data (e.g., "loves hosting parties" + "needs quiet for work")
   - Age + life stage alignment (a 22-year-old fresh grad vs. a 35-year-old professional might score well on paper but be a bad fit)
   - Red flags or contradictions (someone claims cleanliness 9/10 but their bio says "I'm pretty chill about mess")
3. Do NOT re-rank based on things the algorithm already handles well (exact budget math, sleep schedule overlap). Only adjust when you see something the algorithm missed.
4. For the top 5 changes (candidates moved up or down significantly), provide a 1-sentence reason.

Return JSON:
{
  "ranked_ids": ["id1", "id2", ...],
  "adjustments": [
    { "id": "xxx", "moved": "+8", "reason": "Both freelance designers who want a creative live/work space" },
    { "id": "yyy", "moved": "-6", "reason": "Claims to be tidy but bio mentions being relaxed about mess — potential cleanliness conflict" }
  ]
}
```

**Use `claude-sonnet-4-5`** for this — it needs deeper reasoning than Haiku can handle.

## FEATURE 3: FREE-TEXT "IDEAL ROOMMATE" FIELD

### What It Does

Users write a short paragraph describing their ideal roommate or living situation in their own words. Claude parses this into structured signals for matching AND uses the raw text for deeper AI comparisons.

### Database Changes

```sql
ALTER TABLE public.profiles
  ADD COLUMN IF NOT EXISTS ideal_roommate_text TEXT;

ALTER TABLE public.profiles
  ADD COLUMN IF NOT EXISTS ai_parsed_preferences JSONB;
  -- Stores Claude's structured extraction from the free text
  -- Example: {
  --   "vibe": "chill creative",
  --   "schedule_hints": ["works from home", "late riser"],
  --   "social_style": "occasional hangouts, not every night",
  --   "hard_nos": ["loud music after 11pm", "smoking indoors"],
  --   "soft_preferences": ["likes cooking together", "open to sharing groceries"],
  --   "personality_signals": ["introverted", "creative", "neat but not obsessive"]
  -- }
```

### New Edge Function: `parse-ideal-roommate/index.ts`

Called whenever a user saves their `ideal_roommate_text`:

```typescript
// Input: user's free-text description
// Output: structured preference extraction

// Claude Prompt:
`
You are parsing a roommate seeker's description of their ideal living situation. Extract structured preferences from their natural language.

USER'S DESCRIPTION:
"${text}"

Extract the following (leave null if not mentioned):
- vibe: Overall energy/vibe they're looking for (1-3 words)
- schedule_hints: Any mentions of work schedule, sleep, or routines
- social_style: How social they want the household to be
- hard_nos: Things they explicitly don't want (dealbreakers mentioned in text)
- soft_preferences: Nice-to-haves they mention
- personality_signals: Personality traits they describe wanting or having
- cleanliness_hints: Any mentions of tidiness expectations
- noise_hints: Any mentions of noise, quiet, music
- guest_hints: Any mentions of visitors, partners staying over
- budget_hints: Any mentions of price range or financial expectations
- location_hints: Any neighborhoods, areas, or commute needs mentioned

Return valid JSON only. Use short phrases, not full sentences. If the text is too vague to extract anything meaningful, return { "vibe": "flexible", "notes": "Description too vague for structured extraction" }.
`
```

**Use `claude-haiku-4-5-20251001`** — fast extraction task.

### Where Users Enter It

**ProfileQuestionnaireScreen (onboarding):**
- Add as the LAST step before completion
- Header: "Describe your ideal roommate"
- Subtext: "In your own words, what matters most to you? Our AI uses this to find better matches."
- TextInput: multiline, 500 character max, placeholder: "e.g., I'm a grad student who works from home. Looking for someone chill and clean who's okay with my dog. I cook a lot and love sharing meals..."
- "Skip" button — this is optional

**EditProfileScreen:**
- Add a section: "Your Ideal Roommate"
- Same TextInput with current value pre-filled
- When saved, re-trigger `parse-ideal-roommate` edge function to update `ai_parsed_preferences`

**ApartmentPreferencesScreen:**
- Show a condensed version: "Anything else about your ideal living situation?" — shorter prompt, same field

### How It's Used in Matching

1. **AI Match Explanations (Feature 1):** The `ideal_roommate_text` is included in both profiles sent to the explanation generator. Claude can spot alignment between what two people wrote.

2. **AI Deck Re-Ranking (Feature 2):** The full text is included in candidate profiles. Claude uses it for holistic comparison.

3. **Deterministic Algorithm Boost:** The `ai_parsed_preferences` structured data can add up to **5 bonus points** to the existing 100-point score:
   - If `ai_parsed_preferences.hard_nos` from user A conflicts with user B's profile → penalty of -10 points (override, this is effectively a soft dealbreaker)
   - If `ai_parsed_preferences.vibe` keywords match between two users → +2 pts
   - If `ai_parsed_preferences.social_style` aligns → +2 pts
   - If `ai_parsed_preferences.soft_preferences` overlap → +1 pt
   - Cap bonus at +5 total (so max score becomes 105, normalize back to 100 scale)

4. **AI Assistant Context:** When the AI assistant chats with a user, include their `ideal_roommate_text` in the context so it can reference what they said they want.

### Display

- Show the user's `ideal_roommate_text` on their profile card (below bio, above interests)
- Label it: "Looking for..." in a distinct visual style (italic or different background)
- On match cards, if both users have free text, show a small "AI Insight" tag that links to the full explanation

## FEATURE 4: AI HOST MATCHMAKER

### What It Does

For agents and companies on paid tiers, Claude proactively analyzes their listings against the renter pool and generates recommendations: "Here are the 5 best renter groups for your Bushwick 3BR, and here's why."

### New Edge Function: `ai-host-matchmaker/index.ts`

```typescript
// Input: host user ID, listing ID (optional — if omitted, analyze all active listings)
// Output: per-listing recommendations with explanations

// Example output:
{
  listing_id: "xxx",
  listing_summary: "3BR in Bushwick, $4,200/mo",
  recommendations: [
    {
      type: "group",
      id: "group-123",
      members: ["Alice (26)", "Bob (28)", "Carol (25)"],
      match_strength: "strong",
      reason: "All 3 want Bushwick, combined budget $4,500 (7% over listing), 2/3 work from home so they'll appreciate the large living room. Move-in dates align within 2 weeks.",
      action: "Invite to apply"
    },
    {
      type: "renter",
      id: "user-456",
      name: "David (30)",
      match_strength: "good",
      reason: "Wants Bushwick or Bed-Stuy, budget $1,600 (fits if paired with 2 others), no dealbreaker conflicts. Currently forming a group — could pair with your other applicants.",
      action: "Add to shortlist"
    }
  ],
  market_insight: "12 active renters want Bushwick with budgets matching this listing. 3 new renter groups formed this week in the area. Demand is high — consider responding within 48 hours."
}
```

**Claude Prompt Structure:**
```
You are an AI real estate matchmaker for a roommate-finding platform. A host (${host.type}: individual/agent/company) has listings and you need to find the best renter matches.

HOST INFO:
- Name: ${host.name}
- Type: ${host.type}
- Active listings: ${host.listings.length}

LISTING TO MATCH:
- Address: ${listing.neighborhood}, ${listing.city}
- Zip: ${listing.zip_code}
- Bedrooms: ${listing.bedrooms}
- Price: $${listing.price}/month (per person: $${listing.price / listing.bedrooms})
- Available: ${listing.available_date}
- Amenities: ${listing.amenities?.join(', ')}
- Description: ${listing.description}

CANDIDATE RENTERS (pre-filtered by city + budget range):
${renters.map(r => `
  [${r.id}] ${r.first_name} (${r.age})
  - Budget: $${r.budget_max}/mo
  - Neighborhoods: ${r.preferred_neighborhoods?.join(', ')}
  - Move-in: ${r.move_in_date}
  - Ideal roommate: ${r.ideal_roommate_text || 'not set'}
  - Sleep: ${r.sleep_schedule}, Clean: ${r.cleanliness}/10
  - Work: ${r.work_location}
  - In group: ${r.group_id || 'no'}
`).join('\n')}

CANDIDATE GROUPS:
${groups.map(g => `
  [${g.id}] "${g.name}" — ${g.members.length} members
  - Members: ${g.members.map(m => `${m.first_name} (${m.age}, $${m.budget_max})`).join(', ')}
  - Combined budget: $${g.members.reduce((sum, m) => sum + m.budget_max, 0)}
  - Group neighborhoods: ${[...new Set(g.members.flatMap(m => m.preferred_neighborhoods || []))].join(', ')}
  - Move-in range: ${g.earliest_move_in} to ${g.latest_move_in}
`).join('\n')}

INSTRUCTIONS:
1. Recommend the top 5 best matches (groups first, then individual renters who could form groups).
2. For each, explain WHY in 2-3 sentences referencing specific data points.
3. Rate match strength: "strong" (>80% fit), "good" (60-80%), "moderate" (40-60%).
4. Suggest an action for each: "Invite to apply", "Add to shortlist", "Pair with [other renter]", "Send message".
5. End with a 1-2 sentence market_insight about demand in this listing's area.
6. Prioritize:
   a. Groups that fill the listing completely (members = bedrooms)
   b. Budget alignment (combined budget within 15% of listing price)
   c. Move-in date alignment
   d. Neighborhood preference match
   e. Lifestyle compatibility within the group
7. Flag any risks: "One member's budget is 30% below per-person price" or "Move-in dates 2 months apart"

Return valid JSON.
```

**Use `claude-sonnet-4-5`** — complex multi-candidate analysis.

## UPDATE MATCHING ALGORITHM (`utils/matchingAlgorithm.ts`)

Add AI-parsed preference bonus to the scoring:

```typescript
// After all 16 category scores are calculated:

// AI Free-Text Preference Bonus (max +5, normalized back to 100)
let aiBonus = 0;

if (userA.ai_parsed_preferences && userB.ai_parsed_preferences) {
  const prefsA = userA.ai_parsed_preferences;
  const prefsB = userB.ai_parsed_preferences;

  // Check for hard_no conflicts (penalty)
  const hardNoConflict = checkHardNoConflicts(prefsA, prefsB, userA, userB);
  if (hardNoConflict) {
    aiBonus -= 10; // Significant penalty — this is a soft dealbreaker
  }

  // Vibe alignment
  if (prefsA.vibe && prefsB.vibe && vibeAligns(prefsA.vibe, prefsB.vibe)) {
    aiBonus += 2;
  }

  // Social style alignment
  if (prefsA.social_style && prefsB.social_style && socialStyleAligns(prefsA.social_style, prefsB.social_style)) {
    aiBonus += 2;
  }

  // Soft preference overlap
  const softOverlap = countSoftPreferenceOverlap(prefsA.soft_preferences, prefsB.soft_preferences);
  if (softOverlap >= 2) aiBonus += 1;

  // Cap bonus at +5
  aiBonus = Math.max(-10, Math.min(5, aiBonus));
}

// Apply bonus and normalize
const rawTotal = Object.values(breakdown).reduce((sum, v) => sum + v, 0) + aiBonus;
const normalizedTotal = Math.max(0, Math.min(100, Math.round(rawTotal * (100 / 105))));
```

1. **Aggressive caching** — Never regenerate if cache is valid

5. **Usage quotas** — Track in `ai_usage_log`, enforce limits per plan tier
6. **Haiku first** — Use Haiku for everything except re-ranking and host matchmaker (which need Sonnet's reasoning)

## IMPORTANT NOTES

- **AI features are ADDITIVE** — they layer on top of the existing 100-point algorithm. The deterministic scoring is NEVER removed or replaced. AI re-ranking only reorders, it never adds or removes candidates.
- **All AI results are cached** — never call an edge function if a valid cached result exists.
- **Graceful degradation** — if an AI call fails (timeout, rate limit, error), fall back to deterministic results silently. Never show an error to the user.
- **Privacy** — The AI never sees last names, emails, phone numbers, or exact addresses. Only first names, ages, and preference data.
- **Free text is optional** — never block onboarding or profile completion if user skips the ideal roommate field.
- **Token efficiency** — Trim long bios and ideal_roommate_text to 500 chars before sending to AI. Remove null/empty fields from prompts.
- **A/B testing** — Consider showing AI vs non-AI rankings to measure if AI re-ranking actually improves match acceptance rates. Track: swipe-right rate, message rate after match, and eventual lease signing rate for AI-ranked vs deterministic-ranked matches.
