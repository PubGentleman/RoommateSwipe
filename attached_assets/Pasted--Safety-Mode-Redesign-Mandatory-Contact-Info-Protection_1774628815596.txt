# Safety Mode Redesign — Mandatory Contact Info Protection

## What We're Changing and Why

The current Safety Mode is an opt-in toggle in Profile Settings. This means most users never turn it on, so phone numbers and social handles flow freely until someone gets scammed or harassed. The fix is to make contact info protection a platform-level default, with clear unlock paths based on trust events (booking/showing) or a paid subscription.

**New behavior in one sentence:** Free users never see raw contact info — it's always blurred with a clear prompt to book a showing or upgrade. Paid users (Plus/Elite) see contact info freely. All users auto-unlock contact info for a specific conversation once they confirm a showing or booking in that thread.

## New Architecture

### Three states for a conversation:

| User Plan | Showing/Booking Confirmed in This Thread | Contact Info Visible? |
|---|---|---|
| Free | No | ❌ Blurred |
| Free | Yes | ✅ Visible |
| Plus / Elite | Any | ✅ Visible |

### New prop logic in ChatScreen

Replace the single `safetyModeEnabled` boolean with a computed value:

```ts
// At the top of ChatScreen (or in the hook that loads the conversation):

const isPaidUser = renterLimits.canSeeContactInfo; // or hostPlan === 'business'
const [hasBookingInThread, setHasBookingInThread] = useState(false);

// Load booking status for this specific conversation
// Look for a message of type 'showing_confirmed' or 'booking_confirmed' in conversation.messages
// OR query showings/bookings table where conversation_id = currentConversationId
useEffect(() => {
  if (!isPaidUser) {
    const hasConfirmed = messages.some(
      m => m.type === 'showing_confirmed' || m.type === 'booking_confirmed'
    );
    setHasBookingInThread(hasConfirmed);
  }
}, [messages, isPaidUser]);

const contactInfoVisible = isPaidUser || hasBookingInThread;
```

Then in the message render:

```tsx
<SafeMessageText
  text={message.content}
  safetyMode={!contactInfoVisible && message.senderId !== user?.id}
  style={...}
  onUpgradePress={() => navigation.navigate('Plans')}
  onBookShowingPress={() => { /* scroll to or open showing request */ }}
/>
```

## Step 4 — Add Booking Confirmation Detection

**File:** `screens/shared/ChatScreen.tsx`

The booking/showing confirmation can be detected two ways — use whichever matches the existing data model:

**Option A — Message type flag** (if showing requests send a typed message):
```ts
const hasConfirmedEvent = messages.some(
  m => ['showing_confirmed', 'booking_confirmed', 'showing_accepted'].includes(m.type ?? '')
);
```

**Option B — Supabase query** (if showings are tracked in a separate table):
```ts
const { data: showing } = await supabase
  .from('showings')
  .select('id, status')
  .eq('conversation_id', conversationId)
  .in('status', ['confirmed', 'completed'])
  .maybeSingle();

const hasConfirmedEvent = !!showing;
```

Load this in `useFocusEffect` or the main conversation loader. Store as `hasBookingInThread` in local state.
