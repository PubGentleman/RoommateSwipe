# Replit Prompt: Pi Auto-Matching — Automatic Roommate Group Assembly for Agents & Companies

## GOAL

Build an automated system where **Pi assembles roommate groups** without anyone manually swiping or searching. Pi runs on a schedule (nightly or triggered by new signups), analyzes the renter pool, and creates optimized roommate groups based on compatibility score, location preferences, desired group size, and gender preference. Renters are notified first ("Pi found you a match!"). Once renters accept, the confirmed group is packaged and made available to agents/companies on paid plans (Starter+).

## NEW FIELDS TO ADD

### 1. Desired Roommate Count (How Many People They Want to Live With)

This is a **separate field from bedrooms**. Someone wanting a 3BR might want 1 or 2 roommates (one person per room vs. sharing a room). These are independent preferences.

**Add to profiles table:**
```sql
ALTER TABLE public.profiles
  ADD COLUMN IF NOT EXISTS desired_roommate_count INTEGER;
  -- Values: 1, 2, 3, 4 (meaning: I want to live with 1/2/3/4 other people)
  -- NULL means no preference set

ALTER TABLE public.profiles
  ADD COLUMN IF NOT EXISTS desired_bedroom_count INTEGER;
  -- Values: 0 (studio), 1, 2, 3, 4 (meaning: how many bedrooms in the apartment)
  -- NULL means no preference set
  -- NOTE: This is SEPARATE from desired_roommate_count.
  -- Example: someone wants 2 roommates (3 people total) but wants a 4BR (everyone gets their own room + a spare)
```

**Add to ProfileQuestionnaireScreen (optional, post-onboarding) AND EditProfileScreen:**

Field 1 — Roommate Count:
- Label: "How many roommates do you want?"
- Options: 1 roommate (2-person household), 2 roommates (3-person), 3 roommates (4-person), 4+ roommates (5+ person), No preference
- Visual: number selector with icons showing household size

Field 2 — Bedroom Count:
- Label: "How many bedrooms are you looking for?"
- Options: Studio, 1 bedroom, 2 bedrooms, 3 bedrooms, 4+ bedrooms, No preference
- Visual: apartment icons showing layout
- Helper text: "This is about the apartment size, not how many roommates you want"

These are optional — if not set, Pi uses reasonable defaults:
- If only bedroom_count is set (e.g., 3BR) → Pi infers roommate_count as bedrooms - 1 (one person per room)
- If only roommate_count is set (e.g., 2 roommates) → Pi infers bedroom_count as roommate_count + 1 (everyone gets their own room)
- If both are set → Pi respects both exactly

**Add to types/models.ts:**
```typescript
// In Profile or RoommateProfile:
desired_roommate_count?: number; // 1, 2, 3, 4, or null (no preference)
desired_bedroom_count?: number; // 0 (studio), 1, 2, 3, 4, or null (no preference)
```

### 2. Gender Preference for Household

**Add to profiles table:**
```sql
ALTER TABLE public.profiles
  ADD COLUMN IF NOT EXISTS household_gender_preference TEXT;
  -- Values: 'same_gender', 'mixed', 'no_preference'
  -- NULL treated as 'no_preference'
```

**Add to ProfileQuestionnaireScreen (step in the optional section after dealbreakers) AND EditProfileScreen:**
- Label: "Who do you want to live with?"
- Options:
  - "Same gender only" — only matched with people of the same gender
  - "Mixed household is fine" — open to any gender combination
  - "No preference" — default, open to anything
- This should also appear in the Dealbreakers step as an additional option, or as its own mini-step right after dealbreakers

**Add to types/models.ts:**
```typescript
// In Profile or RoommateProfile:
household_gender_preference?: 'same_gender' | 'mixed' | 'no_preference';
```

**Matching logic:**
- If ANY member of a potential group has `same_gender` preference, ALL members in that group must be the same gender
- If a user picks `mixed`, they're explicitly okay with mixed groups
- `no_preference` is treated as open to anything
- The user's `gender` field (male/female/other) determines which "same gender" pool they go into

## THE AUTO-MATCHING PIPELINE

### How It Works (End to End)

```
1. TRIGGER
   ├─ Nightly cron (2 AM local) runs for all active renters not in a group
   ├─ New renter signup (after 24h grace period so they can fill out profile)
   └─ Manual trigger: renter taps "Let Pi find my roommates" button

2. PI ANALYZES RENTER POOL
   ├─ Filter: only renters not already in a confirmed group
   ├─ Filter: only renters active in last 30 days
   ├─ Filter: only renters with at least budget + city set (minimum viable profile)
   └─ Group by: city → desired_roommate_count → household_gender_preference

3. PI ASSEMBLES CANDIDATE GROUPS
   ├─ For each ungrouped renter, find the best 2-4 compatible partners
   ├─ Use the full 100-point matching algorithm across ALL members
   ├─ Factor in:
   │   ├─ Compatibility score (all 16 categories + Pi parsed preferences)
   │   ├─ Location: preferred_neighborhoods overlap + distance
   │   ├─ Group size: desired_roommate_count alignment
   │   ├─ Bedroom count: desired_bedroom_count alignment (group members should agree on apartment size)
   │   ├─ Gender: household_gender_preference compliance
   │   ├─ Budget alignment: combined budget should be realistic for the area AND bedroom count
   │   ├─ Move-in timeline: dates within 30 days of each other
   │   └─ Dealbreaker check: no hard conflicts within the group
   ├─ Score each candidate group as a whole (not just pairwise)
   │   ├─ Group compatibility = average pairwise score across all member pairs
   │   ├─ Minimum pairwise score must be > 50 (no weak links)
   │   └─ Bonus: if all members share a neighborhood preference (+10)
   └─ Select top group per renter (best overall score)

4. PI SENDS TO CLAUDE FOR VALIDATION (Sonnet)
   ├─ Claude reviews the candidate group holistically
   ├─ Checks for subtle conflicts the algorithm might miss
   ├─ Generates a group summary: why these people work together
   └─ Returns: { approved: bool, confidence: high/medium/low, summary: string, risks: string[] }

5. IF APPROVED → NOTIFY RENTERS
   ├─ Create a "pi_auto_group" record with status "pending_acceptance"
   ├─ Send push notification to each member:
   │   "π Pi found your roommates! Meet [Name1], [Name2], and [Name3] — [compatibility]% compatible. Tap to see why."
   ├─ Send email with same message + deep link to the group acceptance screen
   ├─ Each renter sees:
   │   ├─ Pi's group summary (why they were matched)
   │   ├─ Each member's profile card with Pi's Take on compatibility
   │   ├─ Location overlap visualization
   │   ├─ Combined budget range
   │   └─ Accept / Decline buttons
   └─ Renters have 72 hours to accept or decline

6. ACCEPTANCE FLOW
   ├─ All members accept → group status changes to "confirmed"
   │   ├─ Group becomes a real group in the groups table
   │   ├─ Group chat is created
   │   ├─ Group is packaged for agent/company marketplace
   │   └─ Members get: "Your group is confirmed! Agents will reach out with listings."
   ├─ One member declines → group status changes to "partial"
   │   ├─ Remaining members are notified: "[Name] declined. Pi is looking for a replacement."
   │   ├─ Pi searches for a replacement match
   │   └─ New candidate is proposed to remaining members
   ├─ No response in 72h → auto-decline
   │   └─ Group is dissolved, members go back into the pool
   └─ All decline → group is dissolved immediately

7. CONFIRMED GROUP → PACKAGED FOR AGENTS/COMPANIES
   ├─ Group appears in the "Pi Matched Groups" marketplace (new screen)
   ├─ Only visible to agents/companies on Starter+ plans
   ├─ Package includes:
   │   ├─ Group size + member first names + ages
   │   ├─ Combined budget range
   │   ├─ Preferred neighborhoods (aggregated)
   │   ├─ Move-in timeline
   │   ├─ Compatibility score + Pi's group summary
   │   ├─ Amenity must-haves (aggregated)
   │   └─ "Claim Group" button
   └─ Agent claims group → gets access to full profiles + group chat introduction
```

-- Pi Auto-Matched Groups
CREATE TABLE IF NOT EXISTS public.pi_auto_groups (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  status TEXT NOT NULL DEFAULT 'pending_acceptance',
    -- 'assembling', 'pending_acceptance', 'partial', 'confirmed', 'claimed', 'dissolved', 'expired'
  target_size INTEGER NOT NULL, -- desired group size (2, 3, 4, 5)
  target_bedrooms INTEGER, -- desired bedroom count for the group (consensus or inferred)
  city TEXT NOT NULL,
  preferred_neighborhoods JSONB, -- aggregated from members
  combined_budget_min NUMERIC,
  combined_budget_max NUMERIC,

  move_in_latest DATE,
  household_gender TEXT, -- 'male', 'female', 'mixed'
  group_compatibility_score NUMERIC, -- average pairwise score
  min_pairwise_score NUMERIC, -- weakest link score
  pi_summary TEXT, -- Pi's explanation of why this group works
  pi_confidence TEXT, -- 'high', 'medium', 'low'
  pi_risks JSONB, -- any risks Pi identified
  amenity_must_haves JSONB, -- aggregated from members
  created_at TIMESTAMPTZ DEFAULT NOW(),
  acceptance_deadline TIMESTAMPTZ, -- 72 hours from notification
  confirmed_at TIMESTAMPTZ,
  claimed_by UUID REFERENCES public.users(id), -- agent/company who claimed
  claimed_at TIMESTAMPTZ,
  converted_group_id UUID REFERENCES public.groups(id), -- real group after confirmation
  dissolved_reason TEXT -- 'declined', 'expired', 'replaced', 'manual'
);

CREATE INDEX idx_pi_auto_groups_status ON public.pi_auto_groups(status);
CREATE INDEX idx_pi_auto_groups_city ON public.pi_auto_groups(city);
CREATE INDEX idx_pi_auto_groups_claimed ON public.pi_auto_groups(claimed_by);

-- Pi Auto-Group Members
CREATE TABLE IF NOT EXISTS public.pi_auto_group_members (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  pi_auto_group_id UUID REFERENCES public.pi_auto_groups(id) ON DELETE CASCADE,
  user_id UUID REFERENCES public.users(id) ON DELETE CASCADE,
  status TEXT NOT NULL DEFAULT 'pending',
    -- 'pending', 'accepted', 'declined', 'expired', 'removed'

  responded_at TIMESTAMPTZ,

  pi_member_insight TEXT, -- Pi's take on what this person brings to the group

  UNIQUE(pi_auto_group_id, user_id)
);

CREATE INDEX idx_pi_auto_members_user ON public.pi_auto_group_members(user_id);
CREATE INDEX idx_pi_auto_members_group ON public.pi_auto_group_members(pi_auto_group_id);

  listing_id UUID REFERENCES public.listings(id), -- optional: which listing they're matching to
  status TEXT NOT NULL DEFAULT 'claimed',
    -- 'claimed', 'contacted', 'applied', 'placed', 'released'
  claimed_at TIMESTAMPTZ DEFAULT NOW(),
  contacted_at TIMESTAMPTZ,

  UNIQUE(pi_auto_group_id, agent_id)

CREATE INDEX idx_pi_claims_agent ON public.pi_group_claims(agent_id);

## NEW EDGE FUNCTIONS

### 1. `pi-auto-assemble/index.ts` (Main Pipeline)

This is the core function that runs the auto-matching pipeline. Triggered by cron or on-demand.

```typescript
// Input: { city?: string, user_id?: string }
// If city provided: process all ungrouped renters in that city
// If user_id provided: find matches for this specific renter
// If neither: process all cities

// Steps:
// 1. Query ungrouped renters (not in a group, active in 30 days, pi_auto_match_enabled = true)
// 2. Group by city → desired_roommate_count → household_gender_preference
// 3. For each pool, run matching algorithm on all pairs
// 4. Use greedy best-match assembly:
//    a. Find the highest-scoring pair
//    b. If they need more members (desired_roommate_count > 1), find the best 3rd person compatible with BOTH
//    c. Continue until group is full
//    d. Verify: min pairwise score > 50, no dealbreaker conflicts, gender preference compliance
// 5. Send top candidate groups to Claude for validation
// 6. Create pi_auto_groups + pi_auto_group_members records
// 7. Trigger notifications
```

**Claude Prompt for Group Validation:**
```
You are Pi, the AI matchmaker for Rhome. You've assembled a candidate roommate group using compatibility scoring. Now validate whether this group actually makes sense as a household.

GROUP MEMBERS:
${members.map(m => `
  ${m.first_name} (${m.age}, ${m.gender})
  - Occupation: ${m.occupation}
  - Bio: ${m.bio}
  - Ideal roommate: ${m.ideal_roommate_text}
  - Budget: $${m.budget_min}-$${m.budget_max}
  - Neighborhoods: ${m.preferred_neighborhoods?.join(', ')}
  - Sleep: ${m.sleep_schedule}, Clean: ${m.cleanliness}
  - Smoking: ${m.smoking}, Pets: ${m.pets}
  - Work: ${m.work_location}, Guests: ${m.guest_policy}
  - Noise: ${m.noise_tolerance}
  - Roommate relationship: ${m.roommate_relationship}
  - Interests: ${m.interest_tags?.join(', ')}
  - Personality: q1=${m.personality_answers?.q1}, q2=${m.personality_answers?.q2}, q3=${m.personality_answers?.q3}, q4=${m.personality_answers?.q4}, q5=${m.personality_answers?.q5}
  - Desired roommate count: ${m.desired_roommate_count}
  - Desired bedrooms: ${m.desired_bedroom_count}
  - Gender preference: ${m.household_gender_preference}
  - Dealbreakers: ${m.dealbreakers?.join(', ')}
`).join('\n')}

PAIRWISE SCORES:
${pairwiseScores.map(p => `${p.userA} ↔ ${p.userB}: ${p.score}/100`).join('\n')}

AVERAGE GROUP SCORE: ${avgScore}/100
MIN PAIRWISE SCORE: ${minScore}/100

VALIDATE:
1. Does this group make sense as a real household? Consider lifestyle alignment, personality dynamics, and practical factors.
2. Are there any subtle conflicts the scores might miss? (e.g., one person needs quiet for night shifts while another hosts gatherings)
3. Would YOU recommend these people live together?

Return JSON:
{
  "approved": true/false,
  "confidence": "high" | "medium" | "low",
  "summary": "2-3 sentence summary of why this group works (written as Pi, for the renters to read)",
  "member_insights": {
    "[user_id]": "What this person brings to the group (1 sentence)"
  },
  "risks": ["Any potential friction points (0-3 items)"],
  "rejection_reason": "If not approved, explain why (null if approved)"
}
```

**Use `claude-sonnet-4-5`** — this needs deep reasoning about group dynamics.

## PLAN GATING

### Renter Plans (Auto-Match Opt-In)

All renters can opt into Pi auto-matching for free. This is a growth feature — more renters in the pool = better matches = more value for agents.

- **Free/Basic:** Pi auto-match enabled. Limited to 1 pending group invite at a time.
- **Plus:** Pi auto-match enabled. Priority matching (Plus users get matched first). Up to 2 pending invites.
- **Elite:** Pi auto-match enabled. Highest priority. Up to 3 pending invites. Can see Pi's detailed reasoning.

### Agent/Company Plans (Group Access) — Hybrid Model: Plan Inclusions + Pay-Per-Group

All agents/companies can access Pi matched groups. Plans include a monthly allotment of free claims; additional groups are available pay-per-group. This lets smaller agents test the feature without committing to a big plan, while high-volume agents save money on a plan.

**Pay-Per-Group Rate:** $25 per group claim (for any plan, including PayPerUse)

- **PayPerUse ($0/mo):** No free claims included. Pay $25 per group claim. Can browse all groups in their city. Great for testing the feature.
- **Agent Starter ($49/mo):** 3 group claims included/month. Additional claims: $20 each (discounted). Browse all groups in their city.
- **Agent Pro ($99/mo):** 10 group claims included/month. Additional claims: $15 each. Priority notification when new groups match their listings.
- **Agent Business ($149/mo):** 25 group claims included/month. Additional claims: $10 each. Priority notification. Auto-match groups to their listings (Pi proactively sends: "This group is perfect for your Bushwick 3BR").
- **Company Starter ($199/mo):** 10 group claims included/month. Additional claims: $20 each. Browse + filter by all cities.
- **Company Pro ($399/mo):** 30 claims included/month. Additional claims: $10 each. Priority + auto-matching to listings.
- **Company Enterprise (custom):** Unlimited claims included. Exclusive first-look window (see groups 24h before other agents). Custom matching criteria.

**Billing logic:**
- Track `claims_used_this_month` per agent in `pi_group_claims` table
- When claiming: check if `claims_used < plan_included_claims`. If yes, free claim. If no, charge per-group rate.
- Stripe integration: use metered billing for pay-per-group charges
- Show remaining free claims in the UI: "3 of 10 claims used this month. Next claim is free."
- When free claims exhausted: show per-group price before confirming claim: "This claim costs $15. Confirm?"

## IMPORTANT NOTES

- **Opt-in by default** — all new renters have `pi_auto_match_enabled = true`. They can turn it off in settings.
- **One group at a time** — a renter can only be in ONE pending Pi auto-group at a time. If they decline, they go back in the pool.
- **No duplicate matching** — if two renters were already in a dissolved group together, don't re-match them for at least 30 days.
- **Privacy** — agents see limited info before claiming (first names, ages, budget, neighborhoods). Full profiles only after claiming.
- **Gender preference is a HARD filter** — never override it. If someone says "same gender only," that's absolute.
- **Minimum viable profile** — Pi only attempts to match renters who have at least: budget, city, and 1 photo. Encourages profile completion.
- **Pi voice** — all notifications and summaries use Pi's warm, confident, slightly witty persona. Not robotic system messages.
- **Conversion tracking** — track the full funnel: assembled → notified → accepted → confirmed → claimed → contacted → placed. This data is gold for measuring Pi's effectiveness.
