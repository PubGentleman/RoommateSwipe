# Replit Prompt: Pi — Rhome's AI Matchmaker (Explanations, Re-Ranking, Free-Text Preferences & Host Matchmaker)

## GOAL

Transform the matching system from a rigid point calculator into **Pi** — Rhome's AI-powered matchmaker persona. Pi is the intelligence behind every match, recommendation, and insight in the app. The name comes from π (pi) — the mathematical constant that defines relationships between things, just like Pi defines compatibility between roommates.

Four new features layered on top of the existing 100-point deterministic algorithm:

1. **Pi Match Insights** — Pi generates natural language compatibility narratives for every match ("You both love cooking and keep early hours. Heads up — they prefer overnight guests while you don't.")
2. **Pi Deck Re-Ranking** — After the deterministic algorithm scores candidates, Pi re-ranks the top batch by analyzing full profiles holistically, catching nuances the point system misses
3. **Free-Text "Tell Pi What You Want" Field** — Users describe what they want in plain English; Pi parses it into structured signals AND uses it for deeper matching
4. **Pi Host Matchmaker** — For agents/companies on paid tiers, Pi proactively recommends renters and groups for their listings with explanations

## WHAT EXISTS TODAY

### Matching Algorithm (`utils/matchingAlgorithm.ts` — 1010 lines)

100-point weighted scoring across 16 categories:

**High Priority (62 pts):**
- Location (16 pts) — neighborhood/distance matching
- Budget (12 pts) — budget variance scoring
- Sleep Schedule (12 pts) — schedule compatibility
- Cleanliness (12 pts) — cleanliness level diff
- Smoking/Substances (10 pts) — compatibility tiers
- Age (8 pts) — life stage proximity (replaced by move-in timeline below in priority)
- Move-in Timeline (4 pts) — date alignment

**Medium Priority (24 pts):**
- Work Location (6 pts) — remote/office/hybrid
- Guest Policy (6 pts) — overnight guests tolerance
- Noise Tolerance (4 pts) — noise level matching
- Pets (4 pts) — pet ownership/tolerance

**Lower Priority (14 pts):**
- Personality Compatibility (~15% weight) — 5-question quiz results
- Roommate Relationship (2 pts) — social expectations
- Shared Expenses (2 pts) — split preferences
- Lifestyle/Interest Tags (2 pts) — shared interests with conflict penalties
- Zodiac (2 pts) — optional element-based compatibility

**Dealbreaker System:** Hard blocks for `no_smokers`, `no_pets`, `no_overnight_guests`, etc. — these skip scoring entirely and return 0.

### Existing AI Edge Functions (Supabase Deno Functions)

All use Anthropic API — `claude-haiku-4-5-20251001` for fast tasks, `claude-sonnet-4-5` for complex reasoning.

1. **`ai-assistant/index.ts`** (27KB) — Streaming chat with full profile context, AI memory extraction, profile field collection. Daily message limits by plan.
2. **`agent-pair-group/index.ts`** (9.7KB) — AI group recommendations for agents. Takes 2+ renters, recommends best group for a listing. Returns JSON with confidence, reasoning, alternatives.
3. **`calculate-match-scores/index.ts`** (6.3KB) — Pre-calculates simplified scores (budget, sleep, cleanliness, smoking, move-in, pets = 100pts). Runs after swipes. Only re-scores every 24h.
4. **`ai-ask-about-profile/index.ts`** (14.7KB) — AI asks contextual questions to fill profile gaps.
5. **`ai-neighborhood-info/index.ts`** (14KB) — Neighborhood intelligence (transit, vibes, safety, cost).

### AI Memory System

`user_ai_memory` table stores extracted context from conversations:
- Budget stated, move-in timeline, dealbreakers, must-haves
- Preferred neighborhoods, transit preferences
- Work schedule, social preferences
- Free-form memory notes from past conversations

### Discover Service (`services/discoverService.ts` — 335 lines)

Builds swipe deck with filters (budget, room type, compatibility). Applies activity decay scoring and boost rotation. Does NOT use AI for ordering.

### Pre-Calculated Scores (`calculate-match-scores`)

Simpler 100-point score (budget 25, sleep 20, cleanliness 20, smoking 15, move-in 10, pets 10) stored in a scores table. Candidates not scored in 24h get re-scored. Triggers `generate-group-suggestions`.

### Caching & Storage

Create a new table to cache Pi's insights:

```sql
CREATE TABLE IF NOT EXISTS public.pi_match_insights (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_a_id UUID REFERENCES public.users(id) ON DELETE CASCADE,
  user_b_id UUID REFERENCES public.users(id) ON DELETE CASCADE,
  summary TEXT NOT NULL,
  highlights JSONB NOT NULL DEFAULT '[]',
  warnings JSONB NOT NULL DEFAULT '[]',
  confidence TEXT NOT NULL DEFAULT 'medium',
  score_snapshot JSONB, -- the score breakdown at time of generation
  created_at TIMESTAMPTZ DEFAULT NOW(),
  expires_at TIMESTAMPTZ DEFAULT NOW() + INTERVAL '7 days',
  UNIQUE(user_a_id, user_b_id)
);

CREATE INDEX idx_pi_match_insights_users ON public.pi_match_insights(user_a_id, user_b_id);
CREATE INDEX idx_pi_match_insights_expiry ON public.pi_match_insights(expires_at);
```

**Expiry logic:** Regenerate if:
- Explanation is > 7 days old
- Either user updated their profile since the explanation was generated
- The score changed by more than 10 points

### When to Generate

- **Eager generation (background):** When `calculate-match-scores` runs, trigger explanation generation for any match scoring > 60. Store in cache.
- **Lazy generation (on-demand):** If a user views a match card and no cached explanation exists, call the edge function in real-time and cache the result.
- **Never generate** for matches below 40 — not worth the cost.

- Example: "87% Compatible — π Pi's Take: You're both clean early risers who want Williamsburg."

**Group Member Cards:**

## FEATURE 2: PI DECK RE-RANKING

### What It Does

After the deterministic algorithm scores all candidates, the top 30 are passed to Pi for holistic re-ranking. Pi reads full profiles side by side and reorders based on nuances the point system can't capture.

### New Edge Function: `pi-rerank-deck/index.ts`

Create `supabase/functions/pi-rerank-deck/index.ts`:

```typescript
// Input: user profile + array of 30 candidate profiles with scores
// Output: reordered array of candidate IDs with Pi's reasoning

// The function:
// 1. Takes the user's full profile (including ideal_roommate_text, ai_memory, interest_tags)
// 2. Takes top 30 candidates from deterministic scoring
// 3. Sends to Claude for holistic re-ranking
// 4. Returns new order + brief reason for top 5 promotions/demotions
```

**Claude Prompt Structure (Pi's voice):**
```
You are Pi, the AI matchmaker for Rhome — a roommate-finding app. A user is looking for a roommate. You've been given their full profile and 30 potential matches pre-scored by an algorithm.

Your job: re-rank these 30 candidates by TRUE compatibility, catching things the algorithm misses.

THE USER:
- Name: ${user.first_name}, Age: ${user.age}, Gender: ${user.gender}
- Occupation: ${user.occupation || 'not set'}
- Bio: ${user.bio || 'not set'}
- Profile Note: ${user.profile_note || 'not set'}
- Ideal Roommate: ${user.ideal_roommate_text || 'not provided'}
- AI Memory Notes: ${user.ai_memory_notes || 'none'}
- Budget: $${user.budget_min}-$${user.budget_max}/month
- Neighborhoods: ${user.preferred_neighborhoods?.join(', ')}
- Looking for: ${user.looking_for} (${user.desired_bedrooms}BR)
- Lifestyle: sleep ${user.sleep_schedule}, cleanliness ${user.cleanliness}, noise ${user.noise_tolerance}
- Smoking: ${user.smoking}, Pets: ${user.pets}
- Work: ${user.work_location}, Guest policy: ${user.guest_policy}
- Roommate relationship: ${user.roommate_relationship}
- Move-in: ${user.move_in_date}
- Interests: ${user.interest_tags?.join(', ')}
- Diet: ${user.diet_tags?.join(', ') || 'none'}
- Dealbreakers: ${user.dealbreakers?.join(', ')}
- Preferred trains: ${user.preferred_trains?.join(', ') || 'not set'}
- Amenity must-haves: ${user.amenity_must_haves?.join(', ') || 'not set'}
- Shared expenses: utilities=${user.shared_expenses?.utilities}, groceries=${user.shared_expenses?.groceries}
- Personality: after_long_day=${user.personality_answers?.q1}, handle_issues=${user.personality_answers?.q2}, kitchen=${user.personality_answers?.q3}, relationship=${user.personality_answers?.q4}, commute=${user.personality_answers?.q5}

CANDIDATES (ranked by algorithm score):
${candidates.map((c, i) => `
#${i+1} [ID: ${c.id}] Score: ${c.score}/100
  Name: ${c.first_name}, Age: ${c.age}, Gender: ${c.gender}
  Occupation: ${c.occupation || 'not set'}
  Bio: ${c.bio || 'not set'}
  Profile Note: ${c.profile_note || 'not set'}
  Ideal Roommate: ${c.ideal_roommate_text || 'not provided'}
  Budget: $${c.budget_min}-$${c.budget_max}, Move-in: ${c.move_in_date}
  Neighborhoods: ${c.preferred_neighborhoods?.join(', ')}
  Looking for: ${c.looking_for} (${c.desired_bedrooms}BR)
  Sleep: ${c.sleep_schedule}, Clean: ${c.cleanliness}, Noise: ${c.noise_tolerance}
  Smoking: ${c.smoking}, Pets: ${c.pets}
  Work: ${c.work_location}, Guests: ${c.guest_policy}
  Roommate relationship: ${c.roommate_relationship}
  Interests: ${c.interest_tags?.join(', ')}
  Diet: ${c.diet_tags?.join(', ') || 'none'}
  Personality: q1=${c.personality_answers?.q1}, q2=${c.personality_answers?.q2}, q3=${c.personality_answers?.q3}, q4=${c.personality_answers?.q4}, q5=${c.personality_answers?.q5}
  Trains: ${c.preferred_trains?.join(', ') || 'not set'}
  Amenities: ${c.amenity_must_haves?.join(', ') || 'not set'}
`).join('\n')}

INSTRUCTIONS:
1. Re-rank all 30 candidates from best to worst fit for this specific user.
2. Consider — especially the fields the algorithm DOESN'T score:
   - **Bio + Profile Note + Ideal Roommate text** — do their descriptions align? These are the richest signals the algorithm ignores.
   - **Occupation compatibility** — two nurses on opposite shifts = conflict. Two remote freelancers = great fit. A student + a professional might have different lifestyles.
   - **Personality quiz answers together** — someone who decompresses alone (q1=alone) + wants minimal interaction (q4=parallel) is very different from someone who picks alone but wants actual friends (q4=friends). Read the answers holistically.
   - **Amenity must-haves alignment** — two people both needing in-unit laundry and a gym should be prioritized together for group formation.
   - **Transit/commute compatibility** — two people who both take the L train probably want the same neighborhoods even if they picked different ones.
   - **Desired bedrooms** — a 2BR seeker and a 4BR seeker have fundamentally different group size needs.
   - **Diet alignment** — two vegans sharing a kitchen is much smoother than a vegan + someone with no diet preference (not a dealbreaker, but a quality of life signal).
   - **Interest overlap depth** — sharing 1 niche hobby (art, yoga) is more meaningful than sharing 3 generic ones (movies, music, travel).
   - **Potential personality clashes** hidden in the data (e.g., "loves hosting parties" + "needs quiet for work").
   - **Red flags or contradictions** (someone claims very_tidy but their bio says "I'm pretty chill about mess").
   - **Age + life stage alignment** — a 22-year-old fresh grad vs. a 35-year-old professional might score well on paper but be a bad fit.
3. Do NOT re-rank based on things the algorithm already handles well (exact budget math, sleep schedule overlap). Only adjust when you see something the algorithm missed — especially from bio, occupation, personality answers, and apartment preference fields.
4. For the top 5 changes (candidates moved up or down significantly), provide a 1-sentence reason.

Return JSON:
{
  "ranked_ids": ["id1", "id2", ...],
  "adjustments": [
    { "id": "xxx", "moved": "+8", "reason": "Both freelance designers who want a creative live/work space" },
    { "id": "yyy", "moved": "-6", "reason": "Claims to be tidy but bio mentions being relaxed about mess — potential cleanliness conflict" }
  ]
}
```

**Use `claude-sonnet-4-5`** for this — it needs deeper reasoning than Haiku can handle.

## UPDATE MATCHING ALGORITHM (`utils/matchingAlgorithm.ts`)

Add AI-parsed preference bonus to the scoring:

```typescript
// After all 16 category scores are calculated:

// Pi Free-Text Preference Bonus (max +5, normalized back to 100)
let aiBonus = 0;

if (userA.pi_parsed_preferences && userB.pi_parsed_preferences) {
  const prefsA = userA.pi_parsed_preferences;
  const prefsB = userB.pi_parsed_preferences;

  // Check for hard_no conflicts (penalty)
  const hardNoConflict = checkHardNoConflicts(prefsA, prefsB, userA, userB);
  if (hardNoConflict) {
    aiBonus -= 10; // Significant penalty — this is a soft dealbreaker
  }

  // Vibe alignment
  if (prefsA.vibe && prefsB.vibe && vibeAligns(prefsA.vibe, prefsB.vibe)) {
    aiBonus += 2;
  }

  // Social style alignment
  if (prefsA.social_style && prefsB.social_style && socialStyleAligns(prefsA.social_style, prefsB.social_style)) {
    aiBonus += 2;
  }

  // Soft preference overlap
  const softOverlap = countSoftPreferenceOverlap(prefsA.soft_preferences, prefsB.soft_preferences);
  if (softOverlap >= 2) aiBonus += 1;

  // Cap bonus at +5
  aiBonus = Math.max(-10, Math.min(5, aiBonus));
}

// Apply bonus and normalize
const rawTotal = Object.values(breakdown).reduce((sum, v) => sum + v, 0) + aiBonus;
const normalizedTotal = Math.max(0, Math.min(100, Math.round(rawTotal * (100 / 105))));
```

## IMPORTANT NOTES

- **Pi is ADDITIVE** — Pi layers on top of the existing 100-point algorithm. The deterministic scoring is NEVER removed or replaced. Pi's re-ranking only reorders, it never adds or removes candidates.
- **All Pi results are cached** — never call an edge function if a valid cached result exists.
- **Graceful degradation** — if a Pi call fails (timeout, rate limit, error), fall back to deterministic results silently. Never show an error to the user.
- **Privacy** — Pi never sees last names, emails, phone numbers, or exact addresses. Only first names, ages, and preference data.
- **Free text is optional** — never block onboarding or profile completion if user skips the "Tell Pi" field.
- **Token efficiency** — Trim long bios and ideal_roommate_text to 500 chars before sending to Pi. Remove null/empty fields from prompts.
- **Pi's voice** — All user-facing text from Pi should feel like a person talking, not a system. Use "Pi thinks..." or "Pi noticed..." not "The algorithm determined..." Keep it warm, confident, and slightly witty.
- **Pi branding** — Use the π symbol as Pi's icon throughout the app. Badge format: "π Pi Pick", "π Pi's Take", "π Pi Recommended". Color: use the app's accent/brand color for Pi elements.
- **A/B testing** — Consider showing Pi-ranked vs deterministic-ranked decks to measure if Pi actually improves match acceptance rates. Track: swipe-right rate, message rate after match, and eventual lease signing rate for Pi-ranked vs deterministic-ranked matches.
