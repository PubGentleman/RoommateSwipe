# Support System + Safety Mode Host Fix

## Overview of Changes

This prompt covers three things:
1. **Support section** — Replace the broken support UI with a working priority email flow for all user types (renters, agents, companies)
2. **Safety Mode — host side** — Extend the contact info protection logic to cover agents and companies in ChatScreen
3. **Incognito Mode — confirm removal covers all roles** — Make sure no agent or company screen references `hasIncognito` or `incognito_enabled`

---

## Part 1 — Support Section

### The Rule

All users get email support. Paid users get the same email address but their messages are handled with priority. There are two support inboxes:

- **Renters** → `support@rhome.com`
- **Agents and Companies (hosts)** → `hosts@rhome.com`

No live chat. No in-app messaging thread. Just a button that opens the user's email client with a pre-filled subject line that includes their plan tier so your support team can prioritize automatically.

### 1a — Renter Support (ProfileScreen)

**File:** `screens/shared/ProfileScreen.tsx`

Find the current Support section. It likely shows a plain `Alert` with an email address, or an unlinked text block. Replace it entirely.

The support button should call `Linking.openURL` with a `mailto:` link. The subject line includes the user's plan so support staff can prioritize without asking:

```ts
import { Linking } from 'react-native';

const handleContactSupport = async () => {
  const plan = renterLimits.plan ?? 'free'; // 'free' | 'plus' | 'elite'
  const subject = encodeURIComponent(`[${plan.toUpperCase()}] Support Request`);
  const url = `mailto:support@rhome.com?subject=${subject}`;

  const canOpen = await Linking.canOpenURL(url);
  if (canOpen) {
    await Linking.openURL(url);
  } else {
    Alert.alert(
      'Email not available',
      'Please email us at support@rhome.com',
      [{ text: 'OK' }]
    );
  }
};
```

Replace the existing support SettingsItem / button with:

```tsx
<Pressable style={styles.supportButton} onPress={handleContactSupport}>
  <View style={styles.supportButtonLeft}>
    <Feather name="mail" size={18} color="#667eea" />
    <View>
      <ThemedText style={styles.supportButtonTitle}>Contact Support</ThemedText>
      <ThemedText style={styles.supportButtonSub}>
        {renterLimits.plan === 'elite' || renterLimits.plan === 'plus'
          ? 'Priority support — we respond faster'
          : 'We\'ll get back to you as soon as possible'}
      </ThemedText>
    </View>
  </View>
  <Feather name="chevron-right" size={16} color="rgba(255,255,255,0.3)" />
```

### 1b — Agent / Company Support (Host ProfileScreen or Settings)

**File:** `screens/host/ProfileScreen.tsx` or `screens/shared/HostSettingsScreen.tsx` (whichever file renders the host-side profile/settings)

Apply the same pattern but route to `hosts@rhome.com` and use the host's plan:

```ts
const handleHostSupport = async () => {
  const plan = hostPlan ?? 'standard'; // 'standard' | 'business'
  const role = userRole === 'company' ? 'COMPANY' : 'AGENT';
  const subject = encodeURIComponent(`[${role}][${plan.toUpperCase()}] Support Request`);
  const url = `mailto:hosts@rhome.com?subject=${subject}`;

  const canOpen = await Linking.canOpenURL(url);
  if (canOpen) {
    await Linking.openURL(url);
  } else {
    Alert.alert(
      'Email not available',
      'Please email us at hosts@rhome.com',
      [{ text: 'OK' }]
    );
  }
};
```

Render the same `supportButton` UI but with the host-appropriate subtitle:

```tsx
<ThemedText style={styles.supportButtonSub}>
  {hostPlan === 'business' || userRole === 'company'
    ? 'Priority support — we respond faster'
    : 'We\'ll get back to you as soon as possible'}
</ThemedText>
```

Companies always get the priority label since they are always on a Business-equivalent plan.

## Part 2 — Safety Mode Host-Side Fix

### The Rule

The Safety Mode prompt already covers the renter side of `ChatScreen`. This part extends it to cover agents and companies using the same conversation screen.

The contact info visibility logic in `ChatScreen.tsx` currently only checks renter plan limits. It needs to branch based on the logged-in user's role.

---

### 2a — Update `canSeeContactInfo` logic in ChatScreen

**File:** `screens/shared/ChatScreen.tsx`

Find the section added by the Safety Mode prompt that computes `contactInfoVisible`. It currently looks like:

```ts
const isPaidUser = renterLimits.canSeeContactInfo;
```

Replace with a role-aware check:

```ts
const isPaidUser = userRole === 'renter'
  ? renterLimits.canSeeContactInfo          // Plus or Elite renter
  : (hostPlan === 'business' || userRole === 'company'); // Business agent or any company
```

Everything else in the Safety Mode implementation stays the same — the `hasConfirmedEvent` check, the `contactInfoVisible` computed value, and the `SafeMessageText` props. The only change is this one line that determines plan-based visibility.

---

### 2b — Add `hostPlan` to ChatScreen if not already available

If `hostPlan` is not already loaded in `ChatScreen`, read it from the auth context or profile fetch:

```ts
// In the user/profile load effect:
const { data: profile } = await supabase
  .from('profiles')
  .select('role, host_plan')  // or whatever column stores host plan
  .eq('user_id', authUser.id)
  .single();

const userRole = profile?.role;    // 'renter' | 'agent' | 'company'
const hostPlan = profile?.host_plan; // 'standard' | 'business' | null
```

Store both as state variables at the top of the component.

### 2c — Confirm the showing/booking unlock works for hosts too

The `hasConfirmedEvent` detection (checking messages for `showing_confirmed` / `booking_confirmed` types, or querying the showings table by `conversation_id`) already applies to all conversations regardless of role — no changes needed here.

When a showing is confirmed in a thread between an agent and a renter, both sides auto-unlock. The unlock system message (*"🔓 Contact info is now visible in this conversation"*) should appear for both participants.
