## 🔴 CRITICAL BUGS — Things That Are Actively Broken

### 1. AI Edge Function Queries Profiles With Wrong Column
**File:** `supabase/functions/ai-assistant/index.ts` — `getUserProfile()`
**Severity:** Critical — AI never has user context

```ts
// WRONG — profiles.id is a random UUID, not the auth user's ID
const { data } = await supabase
  .from('profiles')
  .select(...)
  .eq('id', userId)   // ← WRONG COLUMN
```

In `migrations/001_schema.sql`, the `profiles` table has its own `id uuid primary key default gen_random_uuid()` and a separate `user_id uuid references public.users(id)`. Querying `.eq('id', userId)` will never find the profile. Should be `.eq('user_id', userId)`.

**Fix:**
```ts
.eq('user_id', userId)
```

---

### 2. `match_scores` Table Does Not Exist in Any Migration
**File:** `supabase/functions/ai-assistant/index.ts` — `getTopMatches()`
**Severity:** Critical — AI always tells users "no top matches yet"

```ts
const { data } = await supabase
  .from('match_scores')   // ← Table does not exist
  .select('target_id, score, profiles!target_id(...)')
  ...
```

None of the 15 migration files (`001_schema.sql` through `015_ai_conversations.sql`) create a `match_scores` table. The query fails silently and returns `[]`, so the AI system prompt always says **"no top matches yet"** for every user, even if they have matches. This is one of the biggest missing pieces.

**Fix:** Need to create this table and a background job or trigger that populates it when matches are created.

```sql
CREATE TABLE IF NOT EXISTS public.match_scores (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  target_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  score INTEGER NOT NULL DEFAULT 0,
  calculated_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(user_id, target_id)
);
CREATE INDEX idx_match_scores_user ON match_scores(user_id, score DESC);
```

### 3. AI Chat Input Buried Under Tab Bar
**File:** `screens/renter/AIAssistantScreen.tsx` + `navigation/RoommatesStackNavigator.tsx`
**Severity:** Critical — chat input not usable

The `AIAssistantScreen` lives inside `RoommatesStackNavigator` as a plain stack screen. The tab bar (`position: absolute, bottom: 0`) sits on top of it. The screen tries to compensate with `keyboardVerticalOffset={TAB_BAR_HEIGHT}` and `paddingBottom: Math.max(insets.bottom, 12) + TAB_BAR_HEIGHT`, but this is a hack that breaks on Android and different inset sizes.

**Fix:** Promote to `fullScreenModal` in `RoommatesStackNavigator.tsx`:
```tsx
<Stack.Screen
  name="AIAssistant"
  component={AIAssistantScreen}
  options={{ presentation: 'fullScreenModal', headerShown: false }}
/>
```
Then remove the manual `TAB_BAR_HEIGHT` padding from the screen itself and use only `insets.bottom`.

### 6. `availableNow` Quick Filter Matches Everything
**File:** `screens/renter/ExploreScreen.tsx` — `applyFilters()`
**Severity:** High — filter chip does nothing useful

```ts
if (activeQuickFilters.has('availableNow')) {
  filtered = filtered.filter(p =>
    p.available === true ||
    (p as any).status === 'active' ||   // ← ALL Supabase listings already pass this
    ...
  );
}
```

Since `getListings()` already filters with `.eq('is_active', true)` and `.eq('is_rented', false)`, every listing already has `status === 'active'`. The "Available Now" chip never filters anything out. It should filter by `available_date <= today` (units that are actually move-in ready today, not future-dated).

**Fix:**
```ts
if (activeQuickFilters.has('availableNow')) {
  const today = new Date();
  today.setHours(0, 0, 0, 0);
  filtered = filtered.filter(p =>
    !p.availableDate || p.availableDate <= today
  );
}
```

### 7. `bestMatch` Chip Has No Filter or Sort Logic
**File:** `screens/renter/ExploreScreen.tsx` — `applyFilters()`
**Severity:** High — the default active chip does nothing

`QUICK_FILTERS` includes `{ key: 'bestMatch', label: 'Best Match' }` and it's in the default active set. But `applyFilters()` has no code that checks `activeQuickFilters.has('bestMatch')`. The sort order runs the same regardless: featured → featured+boosted → boosted → host plan priority.

"Best Match" should sort by compatibility score between the renter and the host. This logic exists (`getPropertyCompatibility()` is defined) but is never used in the sort.

**Fix:** When `bestMatch` is active, add a compatibility-based sort step.

### 8. Cross-Navigator `CreateGroup` Navigation From ExploreScreen
**File:** `screens/renter/ExploreScreen.tsx` — `handleInquireAsGroup()`
**Severity:** High — may crash or silently fail

```ts
navigation.navigate('CreateGroup' as never, {
  listingId: ...,
  existingGroupId: ...,
} as never);
```

`ExploreScreen` is in `RenterTabNavigator`. `CreateGroup` is a screen inside `MessagesStackNavigator`, which is under the `Messages` tab. Navigating directly to `CreateGroup` from a different tab stack using `as never` casts is a pattern that often fails silently or throws a runtime error.

**Fix:** Navigate to the Messages tab first, then to CreateGroup:
```ts
navigation.navigate('Messages' as never, {
  screen: 'CreateGroup',
  params: { listingId: ..., existingGroupId: ... }
} as never);
```

## 🟡 LOGIC ISSUES — Things That Work Incorrectly

---

### 9. AI Screen Has Two Completely Different Response Systems
**File:** `screens/renter/AIAssistantScreen.tsx`

The screen has:
1. **Real path**: `sendAIMessage()` → Supabase Edge Function → Claude API → real contextual response
2. **Fallback path**: `handleSendFallback()` → `generateAIResponse()` → hardcoded keyword-matching → made-up data

The fallback generates things like:
- "Harvest Kitchen - Farm-to-table, $$$"
- "The Velvet Room - Upscale cocktail lounge"
- "Downtown District — Avg rent $1,800–2,500/mo"

These are completely fabricated, city-generic, and misleading. Users have no idea which system responded. If the Edge Function is down or the user isn't authenticated, they get fake responses that look real.

**Fix:** Remove `generateAIResponse()` entirely. If the API fails, show an error state instead of fabricated data.

---

### 10. `loadHostProfiles` in ExploreScreen Reads Mock Data, Not Supabase
**File:** `screens/renter/ExploreScreen.tsx` — `loadHostProfiles()`

```ts
const users = await StorageService.getUsers();  // ← mock local storage
```

Host profiles for compatibility scoring are loaded from `StorageService` (mock data), not from Supabase. So the compatibility percentage shown on listing cards is always calculated against fake mock profiles, never real host data.

---

### 11. `Group.members` Is a Union Type That Causes Fragile Code
**File:** `types/models.ts`

```ts
members: string[] | GroupMember[];
```

This forces defensive code like `typeof m === 'string' ? m : m.userId || m.id` in every place that reads group members. It means the data shape is inconsistent depending on where/how groups are loaded, and it's easy to get a runtime error if the wrong branch is taken.

---

### 12. `subscriptions` Table Default Plan Is `'basic'` But AI Limits Use `'free'`
**File:** `migrations/001_schema.sql` + `supabase/functions/ai-assistant/index.ts`

The subscriptions table defaults new users to plan `'basic'`. The AI Edge Function's `DAILY_LIMITS` has `basic: 5` which works, but the UI and the admin panel treat `'basic'` and `'free'` as the same thing. Inconsistency causes confusion. Recommended: run `UPDATE subscriptions SET plan = 'free' WHERE plan = 'basic'` and remove `'basic'` from the allowed values.

---

### 13. `profilePicture` vs `avatar_url` Data Mismatch
**File:** Throughout codebase

The `User` type uses `profilePicture?: string` but the Supabase `users` table uses `avatar_url text`. The auth context maps between these but inconsistently. In `ExploreScreen`, `user.profilePicture` is used; in the message service, `avatar_url` is read from Supabase. This means profile photos often don't load in certain contexts.

---

### 14. `getConversations()` N+1 Query Problem
**File:** `services/messageService.ts`

```ts
const conversations = await Promise.all(
  matches.map(async (match) => {
    const { data: otherUser } = await supabase.from('users')...   // 1 query per match
    const { data: messages } = await supabase.from('messages')... // 1 query per match
    const { count } = await supabase.from('messages')...           // 1 query per match
  })
);
```

For a user with 20 matches, this fires 60+ Supabase queries. This will be slow and will hit rate limits as the user base grows. Should use a single JOIN query or the `conversations` view defined in the schema.
