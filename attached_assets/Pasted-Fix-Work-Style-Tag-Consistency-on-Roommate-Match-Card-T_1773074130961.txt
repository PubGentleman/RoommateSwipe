Fix Work Style Tag Consistency on Roommate Match Card
There is a data consistency bug on the roommate match card. A user's work style tag (Remote / Office / Hybrid) is showing incorrectly — it can display "Office" while their bio says "working from home." Fix this across the entire app.
The root cause to fix:
The work style tag on the match card must always be pulled from one single source of truth: the workStyle field on the user profile object, set during onboarding. It must never be inferred from the bio text, job title, or any other field. The bio is free text and should never override or conflict with a structured data field.
Specific fixes required:
On the match card, confirm the work style tag reads directly from user.workStyle (or equivalent field name in your schema). The only valid values are "Remote", "Office", and "Hybrid". If the field is null or missing, do not show the tag at all — never show a default or fallback value.
On the onboarding questionnaire, confirm there is a work style question that saves to this field. The question should be: "Where do you work?" with three options: Remote, Office, Hybrid. This answer must be required before the profile is marked complete.
On the profile edit screen, confirm the user can update their work style and that saving it immediately updates the workStyle field — which then reflects everywhere the tag is displayed (match card, profile page, group member chips).
Add a data validation check: when a profile is loaded for display on the match card, if workStyle is "Remote" but the bio contains the word "office," or workStyle is "Office" but the bio contains "work from home" or "remote," log a warning to the console: PROFILE DATA CONFLICT: workStyle tag contradicts bio text for user [userId]. This helps catch future inconsistencies during testing.
Also check these other structured tags on the match card for the same issue: budget (user.budget), neighborhood (user.neighborhood), and move-in date (user.moveInDate). Every tag on the match card must come from a structured field, never inferred from free text. Add the same null-check logic — if any field is missing, hide that tag rather than showing a placeholder or wrong value.
