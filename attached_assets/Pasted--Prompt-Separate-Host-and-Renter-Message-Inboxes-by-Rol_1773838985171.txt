# Prompt: Separate Host and Renter Message Inboxes by Role

## Problem

Both `HostTabNavigator` and `RenterTabNavigator` use the identical `MessagesStackNavigator`, which renders the same `MessagesScreen` with no role-based filtering. When a user switches roles, both inboxes show the same conversations. Host conversations (accepted inquiry chats) bleed into the renter inbox and vice versa.

## Root Cause

Conversation IDs follow distinct patterns that already encode which role they belong to:
- `conv_${matchId}` — renter-to-renter roommate match conversations → **renter only**
- `conv-interest-${id}` — created when a host accepts a renter's interest card → **host only**
- Inquiry threads (`isInquiryThread: true`) — belong to whoever is NOT the host if `hostId !== user.id` → **renter only**, or to the host if `hostId === user.id` → **host only**
- Supabase conversations from `getSupabaseConversations()` — roommate matches → **renter only**

## Step 4 — Update `MessagesScreen.tsx` to read the role param and filter conversations

### 4b. Update the conversation filtering logic inside `loadConversations`

Find this block (around line 171):
```tsx
const userConversations = existingConversations.filter(
  c => !blockedIds.includes(c.participant.id) && (
    c.isInquiryThread ||
    c.id.startsWith('conv-interest-') ||
    c.matchType === 'cold' ||
    matches.some(match =>
      (match.userId1 === user.id && match.userId2 === c.participant.id) ||
      (match.userId2 === user.id && match.userId1 === c.participant.id)
    )
  )
);
```

**Replace it with:**
```tsx
const userConversations = existingConversations.filter(c => {
  if (blockedIds.includes(c.participant.id)) return false;

  if (isHostMode) {
    // Host inbox: accepted interest chats + inquiry threads where user is the host
    return (
      c.id.startsWith('conv-interest-') ||
      (c.isInquiryThread && c.hostId === user.id)
    );
  } else {
    // Renter inbox: roommate match convos + inquiry threads where user is the renter
    return (
      c.isInquiryThread && c.hostId !== user.id ||
      c.matchType === 'cold' ||
      matches.some(match =>
        (match.userId1 === user.id && match.userId2 === c.participant.id) ||
        (match.userId2 === user.id && match.userId1 === c.participant.id)
      )
    );
  }
});
```

## Step 5 — Update inquiry groups loading to be role-aware

The `getMyInquiryGroups()` call at the end of `loadConversations` fetches groups where the user is a renter member. In host mode this should instead fetch groups where the user is the host.

Keep the existing `getMyInquiryGroups()` call for renter mode, and skip it entirely in host mode (host inquiry chats come through the `conv-interest-*` conversations, not through group objects):

```tsx
if (!isHostMode) {
  try {
    const groups = await getMyInquiryGroups();
    // ... existing mapping code ...
    setInquiryGroups(mapped.filter((g: any) => !g.isArchived));
  } catch (e) {
    console.warn('Failed to load inquiry groups:', e);
  }
} else {
  setInquiryGroups([]);
}
```

## What NOT to Change

- `RenterTabNavigator.tsx` — no changes, it keeps using the original `MessagesStackNavigator` (role defaults to `'renter'`)
- `ChatScreen.tsx` — no changes needed, it works for both roles
- `MessagesStackNavigator.tsx` — only the minor type update to `MessagesList` params
- All conversation storage/service logic — no changes
- The `isHostMode` variable is derived purely from the route param, not from `useAuth`, so it always reflects which tab the user is actually in (even if they have dual roles)

## Expected Result

- When user is in **Renter mode**: Messages tab shows roommate match conversations, listing inquiry threads they sent as a renter. No host chats visible.
- When user is in **Host mode**: Messages tab (now labeled "Renter Chats") shows only conversations from accepted interest cards and inquiry group chats where user is the host. No renter match chats visible.
- Switching roles no longer shows the other role's messages.
- Each inbox keeps its own unread count independently.
